zookeeper可靠分布式锁

发表于 2020-12-06

Zookeeper

zookeeper通过维护一个类似文件系统的数据结构，来解决分布式应用中经常遇到的一些数据管理问题，比如：统一命名服务，状态同步，集群管理等等

ZooKeeper的基础知识

  • 持久节点
  • 持久有序节点
  • 临时节点

临时节点是指，client与zk初次连接时，zk会给client分配一个sessionId，然后临时节点的元数据信息中，肯定会带着这个元数据信息的。

临时节点是不允许有子节点的

  • 临时有序节点
  • 容器节点
  • ttl节点

后面两种节点是3.5版本以后支持的。

zk是一种文件系统，可以看作是一棵多叉树，每个节点都有一些元数据信息，可以对zk实现节点的增加删除操作，节点上的数据进行增加删除操作。

节点的元数据信息有版本号的概念，可以实现乐观锁。

Zookeeper实现简易配置中心

假设有A,B两个应用，他们启动时，都会去连数据库的基础配置，现在将基础配置放到zookeeper的某些节点上，然后相应的应用对这些配置节点，设置监听，一旦节点发生了改变，会把改变后的信息推送给客户端。

简单分布式锁实现原理

zookeeper维护一个类似文件系统的数据结构：

每个子目录项下面都被成为目录节点，和文件系统一样，客户端可以自由地添加、删除znode，在一个znode下面增加、删除znode，唯一的不同在于znode是可以存储数据的。

客户端可以注册监听关心的目录节点，当目录节点发生变化(数据改变、被删除、子目录节点增加删除)时，zookeeper通知客户端。

读写分离锁

类似readwriteLock，降低锁的粒度，保证读读不冲突，读写冲突，写写冲突。

考虑到读读不冲突，读写冲突，写写冲突。

每个节点可以去创建目标锁节点

比如

share_lock_read_013123

share_lock_write_021232

如果目标获取读锁，判断前面是否有写锁？没有？则获得锁。有，则设置监听阻塞等待。

如果是获取写锁，判断当前节点是否是第一个节点？是，则获得锁

羊群效应

当我们使用zookeeper实现一个非公平锁的时候，

1000个客户端，抢着创建节点，只有一个节点成功了，剩下999个监听，当获得锁节点释放锁，999个又开始抢锁，。。。。导致时间复杂度过高，羊群/惊群效应

优化版本：

只需要注册比当前节点小的节点即可。

脑裂问题

zookeeper分布式节点采用过半机制实现，分为主从节点，一个master,多个slave。由于zookeeper采用过半机制来保证一致性。部署的时候，比如五台机器，可容忍2台机器宕掉，集群可用。

实际会由于网络原因等，出现假死现象。即部分从节点以为主节点挂掉了，然后再次发生了master节点选举的情况，那么这个时候，就会出现两个主节点的情况。这就是脑裂问题。

脑裂问题的解决方式

  • 法定人数决定

比如3个节点的集群，Quorums = 2, 也就是说集群可以容忍1个节点失效，这时候还能选举出1个lead，集群还可用。比如4个节点的集群，它的Quorums = 3，Quorums要超过3，相当于集群的容忍度还是1，如果2个节点失效，那么整个集群还是无效的。这是zookeeper防止”脑裂”默认采用的方法。

  • 冗余通信

  • 共享资源的方式

Apache Kafka在Exactly-Once Semantics（EOS）上声明了三种粒度的语义保证：

  1. idempotent producer: 单个会话的producer的生产性幂等
  2. transactions: 多会话、多分区的事务保证
  3. 跨读写过程任务的一次精确处理

Kafka 事务

  • 事务id机制

全局transactionId, 即txn.id，由用户提供，故障恢复后也不改变

  • 多会话/多topic/partition事务是怎么做的？
  • 事务协调器如何保证高可用的，挂了咋办

关于数据的强容错性、一致性、存储的容错性方案，采用多副本机制，保证高可用。

min.isr + ack 机制？？？

  • 事务协调器挂了事务的状态怎么恢复的？
    （事务状态如何保证容错性、一致性）
  • producer重启之后，重新abort事务，是怎么实现的？

由于每次重启之后，producer id会被重新生成，所以引入transaction id来解决producer重启之后的事务问题。

几个概念

TransactionCoordinator

  • 处理相关的请求
  • 维护事务的状态信息
  • 向其他的broker发送transaction marker数据

Transaction Log

事务的状态日志信息持久化到内部topic上，__transaction_state。该topic默认有50个partition，transactionId的hash值与分区总数进行hash,得到事务对应的处理分区。

tranaction_state与consumer_offsets一样，是compact类型的topic，换句话说，要求每条消息记录都有key，kafka会针对这个key进行去重合并。

Transaction Marker

当producer完成事务了，进行commit/abort时，需要先告诉事务协调器，再由事务协调器再发送control messages 至各个topic-partition的leader。

也叫control messages，由事务协调器发送出去，用来告诉这个事务涉及到的topic-partition集合的leaders，当前事务操作已经完成了，可以执行commit或者abort操作了。

为啥要当中绕一层呢？而不是直接由生产者去通知到partition的leader呢？

性能考虑

Client 端 事务状态管理

整体的事务状态流程

一. 查找事务协调器

通过事务id的hash值与__transaction_state的partition数量进行取余，确定分区后，然后该分区的leader就是相应的事务协调器

二. 获取PID

请求获取pid，有了pid可以保证事务写入的幂等性。

根据txn.id 进行pid分配，

如果是一个新的事务id, 那么重新进行一个pid的分配，

如果不是新的事务id，那么就获取原有的事务状态信息。

三. 开始事务

本地开始一个事务，注意server端的事务状态转移，与client端的事务状态转移。

四. 执行生产消费处理

比如消费kafka的消息，然后处理一下，完成之后，重新写入到kafka

五. 然后提交或者中止事务

无论是commit/abort事务，对于producer而言，都是向事务协调器发送EndTxnRequest请求，

协调器收到了EndTxnRequest请求之后，会更新事务的元数据信息，然后向该事务涉及到的topic-partition的leader发送control message

最后将这些事务的状态更新为COMMIT或ABORT，并将事务的meta信息持久化到事务日志中。

几个问题

  • 一个txn.id，会不会有多个producer持有？

不会，如果多个 Producer 共用一个 txn.id，那么最后启动的 Producer 会成功运行，会它之前启动的 Producer 都 Fencing 掉

  • txn.id长期不使用，server端怎么处理?

具有事务操作的超时机制，缓存txn.id的相关信息，超过一定的时间期限，将被设置为过期。
