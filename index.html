Apache Kafka在Exactly-Once Semantics（EOS）上声明了三种粒度的语义保证：

  1. idempotent producer: 单个会话的producer的生产性幂等
  2. transactions: 多会话、多分区的事务保证
  3. 跨读写过程任务的一次精确处理

Kafka 事务

  • 事务id机制

全局transactionId, 即txn.id，由用户提供，故障恢复后也不改变

  • 多会话/多topic/partition事务是怎么做的？
  • 事务协调器如何保证高可用的，挂了咋办

关于数据的强容错性、一致性、存储的容错性方案，采用多副本机制，保证高可用。

min.isr + ack 机制？？？

  • 事务协调器挂了事务的状态怎么恢复的？
    （事务状态如何保证容错性、一致性）
  • producer重启之后，重新abort事务，是怎么实现的？

由于每次重启之后，producer id会被重新生成，所以引入transaction id来解决producer重启之后的事务问题。

几个概念

TransactionCoordinator

  • 处理相关的请求
  • 维护事务的状态信息
  • 向其他的broker发送transaction marker数据

Transaction Log

事务的状态日志信息持久化到内部topic上，__transaction_state。该topic默认有50个partition，transactionId的hash值与分区总数进行hash,得到事务对应的处理分区。

tranaction_state与consumer_offsets一样，是compact类型的topic，换句话说，要求每条消息记录都有key，kafka会针对这个key进行去重合并。

Transaction Marker

当producer完成事务了，进行commit/abort时，需要先告诉事务协调器，再由事务协调器再发送control messages 至各个topic-partition的leader。

也叫control messages，由事务协调器发送出去，用来告诉这个事务涉及到的topic-partition集合的leaders，当前事务操作已经完成了，可以执行commit或者abort操作了。

为啥要当中绕一层呢？而不是直接由生产者去通知到partition的leader呢？

性能考虑

Server 端事务状态管理

Client 端 事务状态管理

整体的事务状态流程

一. 查找事务协调器

通过事务id的hash值与__transaction_state的partition数量进行取余，确定分区后，然后该分区的leader就是相应的事务协调器

二. 获取PID

请求获取pid，有了pid可以保证事务写入的幂等性。

根据txn.id 进行pid分配，

如果是一个新的事务id, 那么重新进行一个pid的分配，

如果不是新的事务id，那么就获取原有的事务状态信息。

三. 开始事务

本地开始一个事务，注意server端的事务状态转移，与client端的事务状态转移。

四. 执行生产消费处理

比如消费kafka的消息，然后处理一下，完成之后，重新写入到kafka

五. 然后提交或者中止事务

无论是commit/abort事务，对于producer而言，都是向事务协调器发送EndTxnRequest请求，

协调器收到了EndTxnRequest请求之后，会更新事务的元数据信息，然后向该事务涉及到的topic-partition的leader发送control message

最后将这些事务的状态更新为COMMIT或ABORT，并将事务的meta信息持久化到事务日志中。

几个问题

  • 一个txn.id，会不会有多个producer持有？

不会，如果多个 Producer 共用一个 txn.id，那么最后启动的 Producer 会成功运行，会它之前启动的 Producer 都 Fencing 掉

  • txn.id长期不使用，server端怎么处理?

具有事务操作的超时机制，缓存txn.id的相关信息，超过一定的时间期限，将被设置为过期。

分库分表的总结

发表于 2020-11-19

title: 分库分表的思考
date: 2020-8-10
categories: Java
tags: MySQL

关系型数据库本身易容易成为系统的瓶颈，单机的存储、连接数量、处理能力等，随着业务的不断增长，数据量爆炸，单表500万行或者超过20G，便存在查询性能的瓶颈。

什么是分库分表

切分方式

  • 水平切分

    • 水平切表

就是把一个表的数据，分隔到多个库的多个表中。

优点

不存在单库数据量过大、高并发的性能瓶颈、提升了系统稳定性和负载能力

应用端改造较小，不需要拆分业务模块

缺点

  • 跨分片的事务一致性难以保证

  • 跨库的join关联查询性能较差

  • 数据库多次扩展难度和维护量级大

  • 垂直切分

    • 垂直切库

      根据业务耦合性，将不同的表数据存储在不通的数据库。<类似大系统拆分成小系统>。

    • 垂直切表

      将一个有很多字段的表给拆分成多个表上去。降低数据页大小，效率更高。

优点

垂直切分一定程度提升IO、数据库连接数、单机硬件资源的瓶颈。

缺点

部分表无法join,只能通过接口聚合方式解决，提升了开发的复杂度。

分布式事务处理复杂

依然存在单表数据量过大的问题

数据分片的规则有哪些？

  • 查询分片

将分片路由策略记录在单独的一个库中，每次查询数据库时，先到mapping db查询路由

  • 范围切分

按照id的范围（时间的范围）进行切分

有点冷热数据分离的感觉

这个比较便于扩展，支持范围查询，单表大小可控，但是热点数据可能成为性能的瓶颈。

  • Hash切分

    数据分片比较均匀，集群扩容比较好地避免了这个问题，容易面临跨分片查询的问题。

主键的id问题？

一旦数据库被划分到多个物理节点上，不能依赖数据库自身的主键生成机制。

  • 需要思考全局id唯一的策略
  • 应用程序在插入数据前先获得id，以便进行SQL路由
  1. UUID

uuid标准形式：32个16进制数字，比较占内存，并且建立索引和基于索引查询会存在性能问题。

  1. 结合数据库维护sequence表

创建一张表，

每次需要为某个表的新纪录生成id就从数据库中取出id。

任何插入操作都会访问该表，
该表容易成为性能的瓶颈。

并且存在单点问题，一旦数据库表失效，整个程序将无法工作。

  1. snowflake分布式自增ID算法

生成64位Long型数字，

1(不使用)+41(41bit-时间戳)+10(工作机器id)+12(序列号)

强依赖机器时钟，时钟回拨，可能导致生成的ID重复。

原则

  • 能不切分尽量不要切分

    并不是所有的表都要切分，关注表的数据增长速度。

    不要过度设计、过度优化

    先做力所能及的事情，升级硬件、升级IO、索引优化等等。

缓存一致性讨论

发表于 2020-11-19

缓存更新策略分析

从理论上说，需要给缓存设置过期时间，是保证最终一致性的解决方案。因为有些系统是需要保证性能，可以容忍短时间的缓存与数据库的不一致。

其实，当缓存设置了过期的时间，那么数据库写成功，缓存更新失败，只要到达过期的时间，后面的读请求会自然将缓存加入到新值中去。

先更新数据库，再更新缓存

这个策略是不行的。

从业务角度讲：

  1. 如果缓存是经过一系列计算得到的，那么每次更新缓存前进行大量计算，无疑浪费性能。

  2. 如果业务场景是写多读少，那么缓存总是被更新，也是比较浪费性能的。

从线程安全角度讲：

假设两个线程A,B

A update DB
B update DB
B update cache
A update cache

这种情况就会产生脏数据。

先删除缓存，在更新数据库

这种情况会产生不一致的情况，

比如两个线程A,B

这样就会产生不一致的情况，如果没有设置缓存的过期时间，就会一直产生不一致的情况。
